# Brainstorm

## 概述

使用本技能时，先把用户话题整理成完整提示词，再进入头脑风暴。默认中文输出。

完整提示词必须包含四要素：目标、限定、范围、检验。缺任一项时，不直接思考话题。

本技能只用于讨论、分析、方案设计、评估和决策辅助。即使用户话题里包含“实现”“开发”“落地”“完成”等词，确认结构化提示词后也只输出思考结果，不运行命令、不读取无关代码、不修改文件、不提示或暗示马上进入开发。只有用户在后续消息中主动明确要求开始开发、修改文件、执行命令或提交代码时，才退出本技能流程并进入对应执行流程。

## 四要素

- 目标：用户想得到什么结果，例如方案、列表、决策、架构、风险、命名、路线图、提示词。
- 限定：约束、偏好、禁止项、成本、时间、人力、技术栈、合规、安全或其他边界。
- 范围：讨论覆盖哪些对象、场景、用户、时间、技术、业务线、阶段或不覆盖什么。
- 检验：如何判断输出有用、正确、可执行或可选择，例如可落地、低成本、风险小、三天内能做、可测试、可维护。

## 权限边界

- 读取用户话题、整理提示词、输出分析和方案不需要额外确认。
- 任何会修改文件、运行实现命令、启动服务、提交代码或调用会改变外部状态的操作，都不属于本技能；除非用户在后续请求中明确授权具体执行动作和范围，否则不要执行。

## 执行流程

1. 读取用户话题。
2. 提取并判断是否已有目标、限定、范围、检验。
3. 如果缺失任一要素，停止思考，只要求用户补齐缺失项。
4. 用户补充后，再次检查四要素。
5. 四要素齐全后，先整理为结构化提示词。
6. 请求用户确认结构化提示词。
7. 用户确认后，只开始头脑风暴、分析、方案设计或评估。
8. 输出结尾按“检验”标准做简短自检。

## 缺失处理

- 不要替用户脑补关键约束。
- 可以给候选选项帮助用户收敛，但必须让用户确认。
- 一次最多问 2 到 4 个关键问题。
- 问题要具体，直接对应缺失的四要素。
- 用户说“不确定”时，提供少量可选方向，让用户选择或确认默认值。
- 用户没有确认结构化提示词前，不进入最终头脑风暴。

缺失项回复示例：

```markdown
我先不展开头脑风暴。这个话题还缺少：

- 限定：有哪些约束、偏好或不能做的事？
- 检验：你希望用什么标准判断方案好坏？

请补充这两项。补齐后我会先整理成“目标-限定-范围-检验”，等你确认后再开始思考。
```

## 确认格式

四要素齐全后，使用以下格式让用户确认：

```markdown
我整理的提示词如下，请确认：

目标：

限定：

范围：

检验：

确认后我只开始头脑风暴和方案设计，不执行实现。
```

如果用户修改其中任一项，重新整理并再次确认。

## 思考输出

用户确认后，输出应遵循：

- 围绕目标展开，不偏题。
- 遵守限定，不提出明显违反约束的方案。
- 保持在范围内，不扩散到用户排除的内容。
- 提供多个候选思路时，说明各自适用场景、优点、风险或代价。
- 结尾用检验标准做简短自检，说明哪些方案更符合标准。
- 不主动建议“接下来我可以开始实现”“我会直接推进实现”“确认后进入开发”等执行导向话术；如果需要后续行动，只列为可供用户自行决定的备选方向。

## 禁止事项

- 不要在四要素不完整时直接给方案。
- 不要用“我先假设”绕过用户确认。
- 不要把确认步骤省略为内部思考。
- 不要在用户确认结构化提示词后运行命令、读取实现文件、修改文件、启动服务、运行测试或提交代码。
- 不要把用户对结构化提示词的“确认”解释为允许执行实现。
- 不要提出与限定冲突的建议，除非明确标记为不推荐并解释原因。
