# MyGit Skill

## 概述

在需要把本地改动整理成清晰提交时使用本技能。

目标是：

- 先看清工作区状态。
- 区分本次改动和无关改动。
- 根据仓库实际技术栈选择合适的验证命令。
- 生成聚焦、可读、可追溯的提交信息。

## 权限边界

- `git status`、`git diff`、`git log`、读取文件、检查仓库约定、运行只读分析和起草提交信息不需要额外确认。
- `git add`、交互式暂存、`git commit` 和 `git push` 不视为修改文件；如果代码编写阶段已经确认，且操作范围只包含已确认改动，则不需要再次确认。
- 移动文件、删除文件、回滚、改写历史或任何会改变工作区文件内容的操作，必须先说明操作范围、目标文件和风险，并等待用户确认。
- 暂存、提交和 push 前仍必须复核范围，确认不混入无关改动、未确认改动、密钥、本地配置或构建产物。
- 如果用户在当前请求中已经明确授权具体动作和范围，例如“提交并 push 当前改动”或“直接暂存这些文件”，可以按授权范围执行。

## 提交范围

- 一个提交只包含一个相关主题。
- 不要把无关重构、格式化噪音、功能开发和文档整理混进同一个提交。
- 如果文档是代码变更的必要说明，应与代码放在同一个提交。
- 如果工作区已有用户或其他工具产生的无关改动，先识别并隔离，不要擅自回滚。
- 只暂存本次提交需要的文件；必要时使用 `git add <path>` 或交互式暂存。

## 默认拆分规则

当工作区包含多个可独立审阅、独立回滚的改动时，默认拆分为多个提交，即使它们服务于同一个大主题。不要因为都属于同一个功能方向就合并提交。

遇到以下情况，默认拆分提交，除非用户明确要求合并：

- `docs/`、`summary/`、设计说明、checklist 等文档规划变更，与代码、配置或资源导入同时存在。
- 普通文本文件与 Git LFS 文件、二进制资源或大体积样本资源同时存在。
- 设计文档、实现代码、测试代码、测试资源、构建配置分别代表不同审阅对象。
- 一个改动可以独立 revert 而不破坏另一个改动。
- 提交信息需要同时使用不同 type，例如 `docs` 和 `chore`、`feat` 和 `test`、`build` 和 `docs`。

典型拆分：

- 只新增设计文档：`docs(audio): 增加音频框架设计文档`
- 只导入测试资源：`chore(audio): 添加音频开发测试资源`
- 新增代码实现：`feat(audio): 增加音频播放命令接口`
- 为实现补测试：`test(audio): 覆盖音频 catalog 解析`

如果拆分方案不明显，先列出建议的提交拆分和每个提交包含的路径，再继续暂存和提交。用户要求“提交”但没有明确要求“合并提交”时，仍按本节默认拆分。

如果用户明确要求只提交指定改动、合并为一个提交或不要拆分，优先遵守用户意图；若该要求可能混入无关改动、密钥或高风险文件，先说明风险并按用户确认的范围操作。

## 提交前检查

先运行：

```bash
git status --short
git diff --stat
```

根据仓库实际内容选择验证命令，不要默认假设项目是 Rust。

常见选择：

- Node.js / 前端：`npm test`、`npm run lint`、`npm run build`
- pnpm：`pnpm test`、`pnpm lint`、`pnpm build`
- Python：`pytest`、`ruff check .`、`mypy .`
- Go：`go test ./...`
- Rust：`cargo test`、`cargo check`
- Java / Gradle：`./gradlew test`
- .NET：`dotnet test`
- 文档变更：检查链接、路径、标题、格式和最终内容是否正确

如果仓库已有 README、AGENTS.md、CI 配置、package.json scripts、pyproject.toml、Cargo.toml、Makefile、justfile 或项目文档，先读取相关约定，并优先遵循仓库已有命令。

暂存后、提交前复核暂存区：

```bash
git diff --cached --stat
git diff --cached --name-only
```

确认暂存区只包含本次提交范围内的文件，并检查文件名和 diff 中是否出现 `.env`、证书、私钥、token、密码、数据库连接串或其他本地敏感配置。

如果不能运行某项检查，需要在最终说明或提交正文中明确说明原因。

## 提交信息格式

优先使用：

```text
<type>(<scope>): <summary>

<body paragraph 1>

<body paragraph 2, if needed>
```

scope 没有明显价值时使用：

```text
<type>: <summary>

<body>
```

非交互式提交使用多个 `-m` 参数：

```bash
git commit -m "<type>(<scope>): <summary>" -m "<body paragraph 1>" -m "<body paragraph 2>"
```

常用 `type`：

- `feat`：新功能
- `fix`：问题修复
- `docs`：文档变更
- `refactor`：重构
- `chore`：维护、配置、杂项
- `test`：测试
- `build`：构建系统、依赖、工具链
- `ci`：持续集成
- `perf`：性能优化
- `style`：不影响行为的格式或样式调整

## 提交信息规则

- 默认使用中文。
- 摘要说明实际改了什么，不要写 `update`、`fix bug`、`misc changes` 这类低信息标题。
- 摘要尽量保持一行，简短但具体。
- 保持与仓库历史的大小写、scope 和 type 风格一致。
- 实质性变更应写正文，说明关键模块、行为变化、原因和影响面。
- 跨模块、跨仓库、协议、配置、构建、发布、数据迁移或文档同步类变更必须写正文。
- 正文中可以记录有意义的验证结果，例如测试、构建、lint、格式化、已知警告或未运行原因。
- 很小的改动可以只写一行，例如错字修正、单文件说明更新或机械元数据调整。

## 示例

```text
docs: 整理服务器运维文档结构

将服务器配置移入 docs/ops，并新增根 README、AGENTS 和 docs 索引，明确人类入口、AI 协作规则和文档分层。

补齐 deploy、security、data、observability、capacity、checklists 等目录说明，便于后续按部署、运维、安全和容量规划维护文档。
```

```text
feat(auth): 增加登录限流配置

在登录服务中增加基于 IP 和账号的限流参数，支持通过环境变量调整窗口、阈值和锁定时间。

同步测试用例和配置说明，避免公网登录入口在弱密码或撞库场景下缺少基础保护。
```

```text
fix: 修正 Docker 数据目录权限

修正 /data/docker 初始化权限，确保 Docker daemon 启动后可以正常写入镜像、容器和 volume 数据。
```

## 禁止事项

- 不要提交密钥、密码、token、私钥、数据库连接串或云厂商凭据。
- 不要提交 `.env`、证书、私钥、本地配置或任何包含敏感信息的暂存 diff。
- 不要提交构建产物、缓存目录、日志、临时文件或大体积生成文件，除非仓库明确要求。
- 不要把未经验证的临时代码作为正式提交。
- 不要使用低信息量提交信息。
- 不要为无必要的格式化噪音单独制造提交。
- 不要回滚用户已有改动，除非用户明确要求。
