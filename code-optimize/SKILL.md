# Code Optimize

## 概述

用于对代码做“体检报告”式优化诊断。默认只分析和给出重构路线图，不直接修改代码；只有用户明确要求处理某一个已选问题时，才进入局部重构或代码修改。

目标是找出真实影响维护、扩展、理解、复用或测试的结构性问题。不要输出纯格式化建议。

## 工作模式

根据用户输入选择模式：

- 全局体检：用户给文件、目录、模块或一组代码时，主动审查主要坏味道和结构风险。
- 精准视角：用户指定关注点时，只围绕该视角审查，避免泛泛而谈。
- 单点展开：用户选择报告中的某个问题后，只为这一处生成重构方案或代码修改建议。
- 复审验证：用户已经做过优化后，检查该优化是否降低重复、耦合、复杂度或规范风险。

如果用户没有明确要求修改文件，保持诊断模式，不编辑代码。

## 范围选择原则

结论先行：优先使用“功能模块为主、Git 提交记录为辅”的组合来确定审查范围，尽量避免单纯按文件目录审核。

- 功能模块为主：围绕一个业务能力、功能闭环、核心流程或模块职责选择代码，而不是机械扫描目录下所有文件。
- Git 提交记录为辅：当用户提供本地仓库时，可查看相关文件近期修改频率、反复改动点和历史提交主题，用来识别维护痛点。
- 目录只作为入口线索：如果用户只给目录，先识别其中的功能子模块、核心路径和上下文文件，再分组审查。
- 报告必须说明范围选择依据：写清本次体检按哪个功能模块组织，哪些文件是核心路径，哪些文件只是辅助上下文。

## 诊断流程

1. 确认审查对象：贴入代码、本地文件、目录、模块名或用户指定范围。
2. 先按功能模块和核心流程划分审查边界；如果输入是目录，避免直接按目录平铺审查。
3. 在本地 Git 仓库中，可用提交记录辅助判断热点文件和反复修改的痛点，但不要让提交频率取代结构判断。
4. 读取必要上下文：相关调用方、接口、测试、配置和仓库规则。只读取判断问题所需内容。
5. 识别语言、框架、模块边界和现有风格。
6. 按用户指定视角诊断；如果未指定，执行全局体检。
7. 只报告有实际维护收益的问题；过滤纯格式化、个人偏好和低置信度建议。
8. 每个问题给出位置、类型、优先级、原因、收益、代价和分步路线图。
9. 明确推荐最值得优先处理的 1 到 3 个问题，帮助用户选择后续单点推进。

## 审查视角

全局体检应覆盖：

- 重复逻辑：语义相似但写法不同的逻辑、复制粘贴代码、重复条件判断、重复流程编排。
- 硬编码：魔法值、散落的配置、重复字符串、业务常量没有集中管理。
- 职责不清：类、函数或模块承担多个变化原因，业务、表现、基础设施或平台细节混杂。
- 分层越界：上层直接操作底层细节，底层包含业务策略，模块侵入不属于自己的领域。
- 耦合过高：直接 `new` 具体实现、循环依赖、跨模块访问内部状态、修改一处牵连多处。
- 抽象缺失：多处概念相同但没有统一接口、策略、组件、服务或工具函数。
- 模块边界：公共 API 不清、内部实现外泄、目录职责不稳定、依赖方向混乱。
- 可测试性：核心逻辑难以隔离测试，副作用散落，时间、随机、IO 或外部服务不可替换。
- 命名规范化：名称与真实职责不符，同一概念多个命名，不同概念名称过近，缩写或前后缀风格混乱。
- 注释规范化：注释过期、误导、重复代码表面含义，缺少关键约束、原因、副作用或边界条件说明。

精准视角可使用：

- `重复逻辑与硬编码`
- `职责边界与分层`
- `耦合与依赖`
- `命名规范化`
- `注释规范化`
- `复杂度与可读性`
- `模块抽象与复用`
- `可测试性`
- `性能热点初筛`

## 不报告内容

- 不报告纯格式化建议，例如缩进、空行、括号位置或 import 排序。
- 不报告没有行为、维护或理解收益的个人偏好。
- 不把“可以加注释”“可以改名字”作为独立问题；必须说明当前命名或注释已经造成误解、规范漂移或维护风险。
- 不建议一次性重构所有问题。
- 不在用户只要求诊断时修改代码。

## 优先级

每个问题必须标记优先级：

- `P0`：严重阻碍维护或很容易引入 bug，应优先处理。
- `P1`：明显影响扩展、复用、理解或测试，建议近期处理。
- `P2`：局部可维护性问题，有机会时处理。
- `P3`：轻微规范问题，只适合顺手修改。

优先级必须有理由。不要把轻微命名或注释问题标成高优先级，除非它已经影响边界理解、调用正确性或后续演进。

## 输出格式

使用中文输出诊断报告：

```markdown
## 代码体检范围
- 目标：
- 文件 / 模块：
- 范围选择依据：功能模块为主 / Git 提交热点辅助 / 用户指定文件
- 核心路径：
- 辅助上下文：
- 审查视角：
- 是否修改代码：否

## 总览
- P0：
- P1：
- P2：
- P3：
- 最推荐优先处理：

## 问题 1：<问题标题>
- 优先级：P1
- 类型：重复逻辑 / 硬编码 / 职责不清 / 分层越界 / 耦合 / 抽象缺失 / 命名 / 注释 / 可测试性
- 位置：path/to/file.ext:line
- 现象：
- 为什么是问题：
- 优化收益：
- 风险或代价：
- 推荐重构路线：
  1. 第一步
  2. 第二步
  3. 第三步
- 后续可选动作：适合单独展开为局部重构任务
```

如果没有发现高价值优化项，明确说明“未发现值得单独推进的结构性优化问题”，并列出已检查的视角。

## 单点推进规则

当用户选择某个问题继续推进时：

1. 只处理用户选中的问题，不顺手扩大范围。
2. 先给局部重构方案，包括影响范围、修改步骤、验证方式和回滚风险。
3. 如果用户明确要求修改代码，再执行代码修改。
4. 修改后运行与该问题相关的最小验证；无法运行时说明原因。
5. 不把未选择的问题混入同一次修改。

## 判断标准

- 位置要具体到文件、函数、类或代码片段；能定位行号时给行号。
- 证据要来自代码结构、重复模式、依赖关系、命名冲突、注释内容或测试可达性。
- 收益要具体，例如减少重复、降低耦合、稳定边界、提升可测试性、避免误用。
- 路线图要可分步执行，避免“大重构”“重新设计模块”这类空泛建议。
- 对高风险重构必须说明风险和验证方式。
