# Dev Checklist

## 概述

使用本技能生成或维护通用开发 checklist。checklist 用于把开发需求拆成可独立实现、验证和提交的阶段，帮助后续按阶段推进和提交。

## 基本规则

- 默认使用中文输出。
- 必须要求用户提供输出路径；路径可以是目录，也可以是目标文件。
- 文件名必须使用 `{中文文件名}_checklist.md`。
- 如果用户提供目录和功能名，根据功能名生成中文文件名，例如 `登录功能_checklist.md`。
- 如果用户提供的文件名不符合规则，改为符合规则的文件名，并在执行前说明实际使用的路径。
- 不生成“建议提交拆分”章节；阶段本身就是提交边界。
- 每个阶段应能独立实现、独立验证、独立提交。
- 新建 checklist 时默认使用未完成状态 `[ ]`。
- 只有用户明确说明已完成，或从现有文件、测试结果、提交记录等证据确认完成时，才使用 `[x]`。
- 不臆造开始时间、结束时间、验证结果或完成总结。
- 如果需要定义跨阶段或全项目的整体完成标准，单独新增 `## 最终完成定义`，不要把这些条目拆散到各个阶段。
- 阶段元信息必须紧跟阶段标题，先于所有 checklist 子项；不要把结束时间、总结或验证记录追加到阶段条目之后。
- 每个阶段统一使用“阶段标题、元信息块、空行、checklist 子项”的顺序。

## 权限边界

- 读取需求、读取现有 checklist、查看代码、查看 git 状态、整理计划和生成待写入内容预览不需要额外确认。
- 创建 checklist 文件、写入新 checklist、更新已有 checklist、勾选条目、写入开始时间或结束时间、补充总结和验证记录，都会修改文件，必须先说明目标路径、改动范围和写入内容摘要，并等待用户确认。
- 如果用户在当前请求中已经明确授权具体写入动作和范围，例如“直接创建到这个路径”或“更新这个 checklist 的阶段 2”，可以按授权范围执行，不再额外确认。

## 新建流程

1. 确认用户给出了输出路径和开发主题；缺任一项时先询问。
2. 如果路径是目录，用开发主题生成 `{中文文件名}_checklist.md`。
3. 如果路径是文件，确保文件名符合 `{中文文件名}_checklist.md`；不符合时规范化。
4. 阅读用户提供的需求、相关设计文档、现有代码或仓库约定。
5. 生成目标、基础原则和阶段 checklist。
6. 写入前说明目标路径和内容摘要；未获得明确写入授权时等待用户确认。
7. 写入目标文件；如果文件已存在，转入更新流程，不直接覆盖。

## 更新流程

1. 先读取现有 checklist。
2. 保留已有章节顺序、已勾选状态、阶段时间、阶段总结和验证记录。
3. 根据用户要求或明确证据更新指定阶段。
4. 只追加缺失项或调整明确需要调整的项；不要大面积重写用户已有内容。
5. 如果代码状态、测试结果或用户说法与 checklist 冲突，先说明差异，再按用户确认或证据更新。
6. 写入更新前说明将修改的阶段、条目和元信息；未获得明确更新授权时等待用户确认。
7. 阶段完成时，在阶段标题下方的元信息块补充或更新以下字段，再保留空行并列出 checklist 子项：
   - `开始时间`
   - `结束时间`
   - `开发总结`
   - `验证记录`
8. 如果已有阶段只有 `开始时间` 在子项前，补 `结束时间`、`开发总结` 或 `验证记录` 时必须插入到同一个元信息块中，不要追加到阶段末尾。
9. 如果需要定义跨阶段或全项目的整体完成标准，单独追加 `## 最终完成定义`，并记录：
   - `开始时间`
   - `结束时间`
   - `验收总结`
10. 不要把未验证的任务标为 `[x]`。

## 推荐结构

每个阶段统一使用以下格式：

```markdown
## 阶段 N：阶段标题

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] 阶段任务项
```

新建 checklist 优先使用以下结构，并根据任务规模增减阶段：

```markdown
# 中文功能名 Checklist

## 目标

说明要交付的功能、行为变化、边界和不做什么。

## 基础原则

- [ ] 明确实现范围和非目标。
- [ ] 保持与现有架构、接口、风格和文档约定一致。
- [ ] 每个阶段完成后运行对应验证，并按阶段提交。

## 阶段 1：需求和边界确认

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] ...

## 阶段 2：核心实现

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] ...

## 阶段 3：集成和联动

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] ...

## 阶段 4：测试和文档

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] ...

## 阶段 5：最终验收

- 开始时间：
- 结束时间：
- 开发总结：
- 验证记录：

- [ ] ...

## 最终完成定义

以下项目作为整体完成标准，不要求每个开发阶段都执行，由所有相关阶段完成后统一验收。

- 开始时间：YYYY-MM-DD HH:mm:ss +08:00
- 结束时间：YYYY-MM-DD HH:mm:ss +08:00
- 验收总结：...

- [ ] 最终验收项 1
- [ ] 最终验收项 2
- [ ] ...
```

阶段完成后可扩展为：

```markdown
## 阶段 2：核心实现

- 开始时间：YYYY-MM-DD HH:mm:ss +08:00
- 结束时间：YYYY-MM-DD HH:mm:ss +08:00
- 开发总结：...
- 验证记录：...

- [x] ...
```

## 阶段设计要求

- 阶段标题使用 `阶段 N：标题`。
- 阶段数量按任务复杂度决定，通常 5 到 15 个阶段。
- 如果某个阶段包含多个独立功能、跨模块改动或难以一次验证的工作，应继续拆分，避免单阶段承载过多功能。
- 如果 15 个阶段仍不足以覆盖需求，必须停止生成单个 checklist，先建议拆分为多个 checklist，并说明推荐拆分方式；只有用户明确要求继续生成单文件时，才压缩为一个 checklist。
- 每个阶段聚焦一个可提交开发范围，不把需求确认、核心实现、测试文档和最终验收混在同一阶段。
- 阶段条目必须具体、可验证、可勾选。
- 避免空泛条目，例如“优化体验”“完善逻辑”“处理边界”；改写为能判断完成的任务。
- 对跨模块、数据迁移、配置、兼容性、权限、安全、性能、可观测性和回滚风险单独列项。
- 对 UI 或前端任务加入响应式、交互状态、错误状态和可访问性检查。
- 对后端或接口任务加入输入校验、错误处理、幂等性、权限和日志/指标检查。
- 对工具链或构建任务加入本地命令、CI、缓存、平台兼容和失败路径检查。
- 对游戏或多媒体任务加入资源加载、生命周期、平台、性能和手动验收检查。

## 验证和验收

- 为每个实现阶段列出对应验证项，例如单元测试、集成测试、构建、lint、手动验收或文档检查。
- 验证命令应来自仓库约定；先看 README、AGENTS.md、package scripts、Makefile、justfile、CI 配置或项目文档。
- 如果某项只能手动验收，写清楚验收环境和可观察结果。
- 如果暂时不能验证，保留 `[ ]` 并说明阻塞原因或待验证条件。

## 文件操作约束

- 创建新文件前确认父目录存在；不存在时创建目录。
- 更新已有文件时尽量做局部修改，保留用户已有格式和内容。
- 不创建额外 README、示例文档或模板文件，除非用户明确要求。
- 不把 checklist 写到用户未提供或未确认的位置。
