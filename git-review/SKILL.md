# Git Review

## 概述

用于审查本地 Git 仓库中一段时间内的历史提交，重点发现真实 bug、缺陷、安全风险和高风险测试缺口。默认审查前一天的提交，也可以按用户指定的时间段、分支或 commit 数量审查。

审查结果必须按 commit 组织，并且每个问题必须标记严重程度。

## 权限边界

- 查看 git log、git show、diff、仓库文件和测试配置，以及生成审查报告，不需要额外确认。
- 任何会修改文件或执行修复的操作，都不属于默认审查流程；必须先说明改动范围、目标路径和风险，并等待用户确认。
- 暂存、提交和 push 不视为修改文件；如果修复代码已经确认，且待暂存、提交、push 的范围只包含已确认改动，则不需要再次确认，但必须先复核范围，避免混入无关改动。
- 如果用户在当前请求中已经明确授权具体修复动作和范围，可以按授权范围执行，不再额外确认。

## 工作流程

1. 确认当前目录是目标 Git 仓库；如果用户提供路径，先切到该路径。
2. 读取仓库规则，例如 `AGENTS.md`、项目 README、测试或构建约定。只读取与审查判断相关的内容。
3. 确定审查范围：
   - 默认：前一天 00:00:00 到 23:59:59，使用用户环境本地时区。
   - 用户可指定 `since`、`until`、日期范围、最近 N 个 commit、分支或具体 commit。
   - 默认审查当前分支。
   - 默认跳过 merge commit；如果用户要求包含，则单独标注 merge commit，避免重复报告同一问题。
4. 枚举提交并记录每个 commit 的 SHA、标题、作者、时间和改动文件。
5. 逐 commit 审查它相对父提交引入的 diff，而不是只审查最终工作区状态。
6. 当 diff 不足以判断时，读取必要上下文，例如相关调用方、接口定义、配置、迁移脚本、测试文件或安全边界代码。
7. 只报告高信号问题。不要输出纯风格、命名偏好、格式化建议或没有实际影响的重构意见。
8. 输出报告；没有发现问题的 commit 也要明确标注“未发现高信号问题”。

## 推荐 Git 命令

按时间窗口列提交：

```bash
git log --since="<since>" --until="<until>" --no-merges --date=iso --pretty=format:"%H%x09%an%x09%ad%x09%s"
```

查看单个 commit 统计和改动文件：

```bash
git show --stat --name-only --format=fuller <sha>
```

查看单个 commit diff：

```bash
git show --find-renames --find-copies --format=medium <sha>
```

查看 commit 与父提交：

```bash
git rev-list --parents -n 1 <sha>
```

如果用户指定最近 N 个 commit：

```bash
git log -n <N> --no-merges --date=iso --pretty=format:"%H%x09%an%x09%ad%x09%s"
```

## 审查重点

优先寻找这些问题：

- 正确性：条件判断错误、边界遗漏、空值、异常路径、状态流转错误、异步竞态、返回值或错误码错误。
- 安全：鉴权缺失、越权、注入、路径穿越、敏感信息泄露、日志泄密、不安全默认配置、命令执行风险。
- 数据风险：事务缺失、幂等问题、迁移不兼容、数据丢失、重复写入、缓存和持久化不一致。
- 可靠性：超时、重试、资源释放、连接泄露、错误处理不足、降级缺失。
- 兼容性：API、schema、配置、序列化格式、路径或平台行为的破坏性变化。
- 测试缺口：高风险逻辑缺少必要测试时可以报告，但必须说明具体风险。不要把普通“建议补测试”当作 bug。

## 严重程度

每个问题必须使用以下四级之一，并给出理由：

- `Critical`：可能导致数据破坏、权限绕过、远程利用、生产不可用、密钥泄露或同等级严重后果。
- `High`：很可能触发真实 bug、安全缺陷、兼容性破坏或明显业务错误。
- `Medium`：在特定条件下触发的问题，影响有限但值得修复。
- `Low`：轻微缺陷、边界风险或诊断性不足；仍必须是实际问题，不是风格意见。

如果无法证明问题有实际影响，降级为观察说明或不报告。不要为了填充报告而制造低置信度问题。

## 输出格式

使用中文输出。报告应包含：

```markdown
## 审查范围
- 仓库：
- 分支：
- 时间段：
- 提交数：
- 策略：

## 结果概览
- Critical：
- High：
- Medium：
- Low：
- 未发现高信号问题的提交：

## Commit: <short-sha> <title>
- 作者：
- 时间：
- 文件：

### 问题 1
- 严重程度：High
- 类型：正确性 / 安全 / 数据风险 / 可靠性 / 兼容性 / 测试缺口
- 位置：path/to/file.ext:line
- 问题：
- 影响：
- 证据：
- 修复建议：
- 验证建议：
```

如果某个 commit 未发现问题：

```markdown
## Commit: <short-sha> <title>
- 结论：未发现高信号问题。
```

## 判断规则

- 优先报告由该 commit 引入或暴露的问题；不要把历史遗留问题归咎于当前 commit，除非当前 commit 扩大了影响面。
- 对每个 finding 明确说明证据来自 diff、上下文文件、测试结果还是推理。
- 如果行号来自 diff 之外的上下文，说明这是上下文位置。
- 对安全问题优先说明可利用条件、受影响入口和泄露或越权后果。
- 对兼容性问题优先说明受影响调用方、旧数据、旧客户端或部署顺序。
- 对测试缺口，只有在能指出具体高风险路径时才报告。
- 最终不要自动修改代码；除非用户明确要求修复，否则只做审查和建议。

## 无提交或无法判断

- 如果时间窗口内没有提交，明确说明范围和“未找到提交”。
- 如果仓库不是 Git 仓库，要求用户提供正确仓库路径。
- 如果 diff 过大，先按 commit 和文件风险分组，优先审查认证、权限、数据、迁移、配置、并发和公共 API 改动。
- 如果缺少运行环境或测试无法执行，在报告中说明未执行原因，不要把未验证结果写成确定事实。
